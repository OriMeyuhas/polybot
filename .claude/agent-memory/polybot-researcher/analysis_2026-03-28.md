## Findings (2026-03-28)

### 1. PnL Settlement Excludes Fees (CRITICAL)
Position.profit_if_up() = up_qty * (1 - avg_up) - dn_cost. The avg_up = up_cost/up_qty INCLUDES fees (compute_fee added during fill).
But settlement revenue is up_qty * $1.00 (share pays $1). The REAL revenue is up_qty * $1.00 MINUS the redemption fee.
Polymarket charges fees on redemption too. The bot's PnL is overstated.
**Why:** Real PnL will be lower than paper PnL by ~1-2% of position value.
**How to apply:** Before going live, verify Polymarket redemption fee structure and subtract from settlement PnL.

### 2. Risk Manager is Minimal
Only: drawdown circuit breaker (daily PnL), position count limit, timing gate (no_trade_final_sec).
Missing for live: per-window loss limit, max capital-at-risk, consecutive loss tracking, spread between filled UP/DN qty guard.
**Why:** A series of bad windows could deplete bankroll before daily drawdown triggers.
**How to apply:** Propose adding per-window max loss and consecutive loss tracking.

### 3. Redemption Not Implemented
_redeem_tokens() logs CRITICAL and returns 0.0. In live mode, winning tokens sit unredeemed.
Capital is locked until manual redemption. This blocks compounding.
**Why:** Without auto-redemption, live bankroll shrinks with every window (capital locked in unredeemed tokens).
**How to apply:** Implement CTF Exchange merge/redeem call, or use Polymarket's redemption helper contract.

### 4. One-sided Ladder Bug (run_v4 evidence)
First ladder posted: "15 UP rungs + 0 DN rungs | pair_cost=0.000". Budget split produced 0 DN rungs.
This creates pure directional exposure — no pair cost benefit.
**Why:** Dynamic budget skew can allocate near-zero to one side when asks are very asymmetric (e.g., UP=0.25, DN=0.75 -> DN gets ~$1.65 budget, not enough for meaningful rungs after the cheap-side-heavy allocation inverts it).
**How to apply:** Add minimum budget floor per side (at least MIN_ORDER_SIZE * 2).

### 5. Paper Fill Simulation Biases Results
PaperClobClient.tick() fills 1 order per token per tick (500ms). Real exchange fills can be partial, multiple, or never.
Paper mode also uses midpoint price as fill trigger — real fills happen at the posted limit price against taker flow.
**Why:** Paper PnL is systematically different from live PnL. Cannot trust paper results for live sizing decisions.
**How to apply:** Note this limitation; don't over-optimize on paper results. Live mode fill detection via reconcile() is correct.
